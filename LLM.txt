Generate a compact adaptive per file LLM index for a Go module.

Goal

Minimize tokens while preserving all non default, behaviorally relevant information present in each file.
Use semantic packing: merge homogeneous facts when no information is lost.
Treat the defaults below as a compression dictionary: never restate facts implied by defaults.

Hard rules

English only.
Plain text only.
Output must avoid characters commonly interpreted by Markdown.
Do not use backticks.
Do not use markdown headings, emphasis, tables, block quotes, code fences, list markers, or horizontal rules.
Prefer letters, digits, spaces, commas, periods, semicolons, and parentheses.
One output block per go file.
Each file index must be emitted as a separate standalone plain text block so the client UI can attach a copy button per block if supported.
Extremely concise, telegraphic style.
Facts first. Infer only when strongly supported.
Preserve all non default facts; compress wording, not information.
If two outputs preserve the same non default facts, choose the shorter one.
If unknown and important: Unknown.
If absent or default: omit section entirely.
Do not hallucinate.
Omit standard library imports.
Testing stack: Uber mockgen, testify.
Do not expand obvious Go syntax into prose when it adds no behavioral information.
Group homogeneous declarations into one fact line whenever no information is lost.
Do not mirror the AST.
Do not emit warnings for merely theoretical misuse; warn only on real hazards evidenced by code.

Adaptive output size

Tiny file: ultra short index, usually 3 to 6 lines.
Medium file: normal compact index, only non default sections.
Complex file: full compact index, still only non default sections.

Complexity heuristic

Tiny: 0 to 2 non trivial funcs or methods, little or no state, little or no I O, no concurrency, no lifecycle, no important globals; or declaration only file.
Medium: several important funcs or methods or meaningful state, I O, config, dependencies.
Complex: service, handler, transport, repository, bootstrap, core domain file; multiple contracts; lifecycle, concurrency, security, operational significance.

Default assumptions

Omit any aspect that matches these defaults.

Information preservation rules

Keep every explicit non default fact that affects API, validation, configuration, side effects, lifecycle, security, I O, concurrency, or operational behavior.
Shared facts may be stated once for a group of symbols.
Do not split one semantic fact into multiple lines if one line preserves it fully.
For declaration only files, preserve distinctions only when they matter behaviorally.
If a symbol name fully conveys its role and the file adds no extra behavior, listing the name may be sufficient.
Preserve exported API surface, but it may be represented as grouped summaries rather than one entry per symbol when no non default information is lost.
Distinguish symbols separately only when representation, constraints, effects, or role differ in a behaviorally relevant way.

Inclusion rules for symbols

Always include exported funcs or methods if they define real API or behavior.
Exported types, constants, and vars may be grouped and summarized when no non default information is lost.
Include unexported symbols only if they contain important logic, state mutation, I O, lifecycle, concurrency, security checks, registration side effects, or are central to understanding the file.

Group constants or vars when they share domain, type role, or value pattern.

Use only sections that contain non default useful information.

Do not use characters commonly interpreted by Markdown if a plain sentence can express the same fact.
Prefer comma separated or semicolon separated packing.

Tiny file compression rules

If a file contains only constants, aliases, newtypes, or passive structs with no methods, prefer 3 to 5 lines total.
Group same purpose constants on one line.
Group same shape types on one line.
Omit Functions section entirely if no behavior exists.
If Purpose subsumes most declarations, prefer a short grouped summary over per symbol entries.
Do not restate sizes, primitive kinds, tags, or array lengths per symbol when shared by all listed symbols; state once.
If the source is shorter than the prospective index, compress further.

Medium file mode

Use minimal header plus only non default sections needed for behavior, globals, key functions, and risks.

Complex file mode

Use all non default sections needed to capture API, contracts, state, dependencies, concurrency, lifecycle, security, operations, and key call flow.

Normalization rules

Zero Value values, when included: usable, not usable, or partially usable.
Thread Safety values, when included: yes, no, or conditional.
Production Readiness values, when included: ready, mostly ready, partial, or risky.
Prefer grouped forms such as A, B, C: shared fact or A slash B slash C: shared fact.
State a shared numeric or property value once when it applies to all listed symbols.
If a fact is obvious from signature and adds no semantic value, omit it.
Prefer omission over writing None.
Prefer one compact line over multiple lines when clarity is preserved.

Anti bloat rules

Do not explain what is directly obvious from Go syntax unless it changes behavior.
Do not enumerate symbols just because they are exported.
Do not convert one obvious declaration into multiple prose facts.
Do not state negative or default facts to complete a schema.
Omit any fact recoverable from defaults plus a shorter grouped statement.
When in doubt, shorten.

Quality target

The index must be the shortest text that preserves all non default, behaviorally relevant information from the file.
For tiny files, prefer aggressive semantic packing.
A correct grouped summary is better than a longer per symbol inventory.
The index may be shorter than the source and should often be.
